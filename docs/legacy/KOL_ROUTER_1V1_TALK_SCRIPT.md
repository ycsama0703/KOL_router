# KOL Router 1v1 Talk Script

用途：和老师一对一过项目。语气尽量自然，不按正式答辩腔写。

## P1. 一句话

老师，我这个项目想解决的是金融社媒信息太多的时候，agent 到底该先看什么。

核心发现是：KOL 的社媒流里不是完全乱的，里面有一个比较稳定的“谁更常发起新叙事”的结构。  
我不是想做一个更大的模型，而是想把这个结构做成一个很轻的前端 router，在叙事还没扩散之前，先判断哪些候选内容值得送进后面的 LLM、memory 或 RAG。

所以一句话就是：先用 KOL 数据里的起源结构做低延迟筛选，再让昂贵模型处理更小、更有价值的候选集。

## P2. 现状问题

现在的问题不是我们没有社媒数据，而是数据太多了。

比如一个 symbol 一天里，会有很多 KOL 都在讲不同的故事。每个故事看起来都可能有道理，但 agent 不可能把所有东西都放进 memory 或者全部交给 LLM。

如果等转发、follower、二次传播都出现以后再判断，那其实已经晚了。那个时候我们知道它热，是因为它已经扩散了，不是 early routing。

所以我的问题设定很前置：在一个叙事刚被某个 KOL 第一次提出的时候，我们能不能判断它值不值得进入后面的推理链路。

## P3. 我们发现了什么

这里我想强调一点：我不是在给 KOL 打一个静态好坏标签。

真正的问题不是“这个 KOL 好不好”，而是：这个新说法，由这个历史角色的人，在这个上下文里提出，是否值得进入下游推理。

也就是说，信号有三块：

第一，当前出现的是一个新的叙事候选；  
第二，这个候选是谁先提出来的，这个人历史上是不是经常站在叙事扩散链前端；  
第三，它出现时的上下文是什么，比如时间、可见度、新颖性。

所以这个结构不是账号排行榜，而是“新叙事 × 发起者历史角色 × 当前上下文”的关系。

## P4. 结构是否普遍存在

接下来我做了一步比较基础但很重要的事情：确认这不是个别案例。

数据规模上，我们有 3380 万条 KOL tweet，工作面板里是 45.9 万条，覆盖 3712 个 KOL 账号和 17 个交易标的。

我主要验证了几件事：

第一，KOL 之间确实有 lead-lag hierarchy，不是随机顺序。  
第二，这个结构不是 follower 大小造成的，因为历史源头分数和 follower 的相关性很低。  
第三，它也不是谁时区更早、谁醒得早造成的，校正发帖时间以后这个问题基本消掉了。  
第四，这个结构跨时间、跨资产都有稳定性。

所以这里的结论是：KOL 流里确实存在一种可重复观察到的叙事起源结构。

## P5. 方法

方法本身其实很简单，我故意没有做复杂神经网络。

我们给每个候选叙事一个 score。这个 score 里面有普通上下文特征，比如 visibility、novelty、timing；另外加入一个核心变量，就是这个发起者的历史源头分数。

然后再加两个交互项：历史源头分数和 visibility 的交互，历史源头分数和 novelty 的交互。

直觉上就是：一个人历史上经常发起后续被接续的叙事，这是一种先验；但它只有和当前内容的新颖性、可见度、时点结合，才真正有用。

这里卖点不是模型复杂，而是反过来：如果这么简单的 ridge 都能有效，那说明价值主要来自我们发现的数据结构，而不是模型容量。

## P6. 实验协议

实验里我比较注意 point-in-time，避免事后信息泄露。

也就是说，模型在某个时间点做判断时，只能用那个时间点之前能看到的信息。后面谁转发了、谁接续了、最终热度是多少，这些只能当 label，不能进 router。

候选集的构造是这样的：在一个事件里，如果某个 KOL 第一次提出一个语义上新的 frame，我们就把它作为候选。这里主实验用的是 semantic threshold 0.55，origin window 是 first10，也就是只看早期候选。

label 是 future follower-weighted reach，简单说就是：后续有多少有影响力的 KOL 加入了同一个叙事。我们用这个来评估排序质量。

所以这页主要想说明：大样本发现结构，PIT split 检验这个结构能不能真的在线使用。

## P7. Benchmark 和指标

对比组我没有只挑弱 baseline，而是分了几类。

有最基础的 scale baseline，比如 follower；有不含 OL 的强上下文 baseline；有 surface text；有 BERT、Qwen 这类 embedding；也有本地 LLM 和商业 API 模型。

指标上，最主要看 NDCG@3。它回答的是：模型排在前 3 的候选里，是不是把未来 reach 高的叙事排到了前面。

Hit@1 看第一名是不是命中最强候选。  
Mass@3 看 top 3 捕获了多少未来 reach。  
JS 看模型给出的注意力分布和真实未来 reach 分布接不接近，越低越好。  
Latency 看在线打分阶段的额外延迟。

统计上，核心比较用 paired bootstrap，避免只看一个点估计。

## P8. 主实验结果

主实验结果里，OL-Origin 是最强的一行。

它的 NDCG@3 是 0.755，Hit@1 是 0.493，JS 也是最低的 0.263。  
相比 No-OL Strong，NDCG 提升是 +0.043，Hit 提升是 +0.072。

这里比较重要的是，它不是只赢弱 baseline。它也和 Qwen3-4B embedding、GPT、Claude 这些强对比放在一起比。

Qwen3-4B 已经很强，NDCG 是 0.750；但 OL-Origin 还是略高，而且延迟几乎可以忽略，在线每个 query 约 0.003 ms，也不需要 prompt token。

所以这页我想传达的是：结构化 router 不只是便宜，它在这个任务上也很有竞争力。

## P9. 消融实验

消融这里主要回答：到底是不是历史源头结构在起作用。

先看 OL Only，只用历史源头分数，效果并不好。这说明不能简单说“某些 KOL 厉害”，单独用账号历史不够。

No-OL Strong 加了 rank、time、sentiment、novelty、history 这些上下文，但不加源头分数，已经比较强。

Full OL-Origin 在这个基础上加入校正后的历史源头分数，以及它和 visibility、novelty 的交互，效果最好。

中间还有两个 sanity check：一个是把 OL 分数打乱，一个是用 follower 替代 OL channel。它们的提升都弱于 full model。

所以消融的结论是：历史源头分数必须和当前上下文结合；而且这个信号不是简单的大号效应。

## P10. Agent 小实验

前面主实验是 ranking benchmark。这里我想更贴近真实 agent 使用场景。

设定是：每个 symbol-day 有 30 个候选早期叙事。Full LLM 是直接把 30 个全读进去，让它选；OL-Origin 是先用本地 router 选出 10 个或 20 个，再交给同一个 LLM 做 reranking。

结果比较有意思：先路由再给 LLM，反而比 Full LLM 直接看全部候选更好。

比如 DeepSeek 这组，Full capture 是 0.184，OL b=10 可以到 0.315，提升 71%。  
Claude、Gemini、GPT 也是类似方向。

我的理解是：LLM 面对大量社媒候选时，不一定真的会用好这些数据。前端 router 先压缩候选集，反而能让后面的 LLM 更专注。

延迟上也很明显，Full selector 用 LLM 自己做选择，是秒级；OL selector 是本地 router，是 0.048 ms 量级。

## P11. 总结

最后我想把这个项目压缩成四句话。

第一，问题是 agent 面对高频社媒时，瓶颈在 first-stage attention allocation，也就是先看什么。

第二，我们发现 KOL 流里有一个稳定的叙事起源结构。这个结构不是 follower、时区、频率这些简单解释。

第三，这个结构可以做成一个很轻的前端 router，先决定哪些新叙事值得进入 LLM、memory 或 RAG。

第四，实验上我们用了大样本 KOL panel、PIT split、强 benchmark、消融实验和 agent 场景小实验来验证。

所以我的主线不是“我做了一个更复杂的模型”，而是：如果大样本里发现的 KOL 结构，用简单 ridge 就能有效路由，那说明真正有价值的是这个结构本身。

## 老师可能追问时的短答

**这是不是给 KOL 打标签？**  
不是。标签只是一个历史先验，最后判断的是“某个新叙事在当前上下文里是否值得进下游推理”。

**为什么不用更复杂模型？**  
这篇文章的卖点不是模型容量，而是数据结构。简单模型有效，反而更能说明结构本身有价值。

**会不会只是 follower 效应？**  
我们做了 follower baseline 和 follower replacement，历史源头分数不是 follower 的简单替代。

**会不会是时区效应？**  
raw lead 和发帖时间确实相关，所以我们做了时间校正；校正后相关性基本消掉。

**Qwen 很强，为什么还需要 OL router？**  
Qwen 是强 benchmark，但它需要文本 embedding 和更高延迟。OL router 的优势是本地、低延迟、零 prompt token，而且解释性更强。

**这个东西真实 agent 里怎么用？**  
作为第一层筛选器。先把大量早期候选压成短名单，再交给 LLM 或 memory，而不是让 LLM 直接读全部社媒流。
